# Smarteval Context for AI Coding Agents

This document gives an AI coding agent enough product and engineering context to work on **Smarteval** without re-discovering the concept from scratch.

It is intended to be used together with `smarteval_claude_code_codex_integration_plan.md`, which covers how Smarteval should integrate with Claude Code, Codex, agent skills, CLI workflows, MCP, and repository-level agent instructions.

This file focuses on the **what** and **why** of Smarteval. The integration plan focuses on the **how agents should use it**.

---

## 1. Product summary

**Smarteval** is a project-agnostic evaluation and improvement framework for non-deterministic AI system components, especially components powered by LLMs, image-generation models, RAG pipelines, and agent workflows.

The core idea is:

> Point Smarteval at a probabilistic AI component, define what “good” means, let it generate an eval harness, run controlled experiments, propose improvement hypotheses, and produce an auditable report showing what improved and why.

Smarteval should help engineers move from ad hoc prompt tweaking to a repeatable, evidence-based improvement process.

---

## 2. Problem being solved

Modern AI-enabled applications often contain components whose behavior is non-deterministic or difficult to validate with classic unit tests.

Examples:

- LLM-generated text
- Summaries
- Classification using an LLM
- Entity extraction
- RAG answers
- Image-generation prompts
- Agentic workflows
- Tool-selection behavior
- Structured outputs generated by a model

Engineers often improve these systems by manually changing prompts, model settings, examples, or retrieval configuration. This creates several problems:

- Improvements are subjective.
- Regressions are easy to miss.
- Prompt changes are rarely tested systematically.
- Teams lack a repeatable evaluation loop.
- Cost, latency, format adherence, safety, and quality are not compared consistently.
- Final decisions are often based on a few cherry-picked examples rather than a broader dataset.

Smarteval exists to turn this process into a structured loop.

---

## 3. Core positioning

Smarteval can be described as:

> Test-driven development for prompt-driven and LLM-driven systems.

A more precise positioning statement:

> Smarteval is an agentic evaluation and optimization framework for non-deterministic AI system components. It interviews the engineer, designs the eval strategy, generates the harness, runs controlled experiments, proposes improvement hypotheses, and produces an auditable report showing which changes improved the system and why.

### Critical positioning guardrail

Smarteval should not try to become a generic eval platform, hosted observability product, or benchmark suite. Existing tools already cover large parts of that space.

The strongest product wedge is narrower:

> Smarteval is the repo-local evaluation loop that AI coding agents can use before, during, and after changing AI behavior in an application codebase.

This matters because engineers working with agents need more than eval mechanics. They need a workflow that prevents the agent from making unmeasured prompt, model, retrieval, or agent-instruction changes.

### Ideal first users

The first users should be engineers who:

- already use Claude Code, Codex, Cursor, or similar AI coding agents;
- own application code with LLM, RAG, image-generation, structured-output, or agentic behavior;
- want evidence before accepting prompt or model changes;
- prefer local repo artifacts over a new hosted workflow;
- need reports that can be reviewed in pull requests.

Smarteval is a weaker fit for teams that already have a mature eval platform, production tracing, dataset management, and release gates unless Smarteval can integrate with that stack instead of replacing it.

### Non-goals for the MVP

The MVP should not include:

- hosted dashboards;
- team account management;
- production monitoring;
- broad multimodal eval support;
- automatic production patching;
- autonomous prompt optimization without review;
- complex search or optimization algorithms;
- mandatory MCP or CI integration.

The MVP should prove that Smarteval is easier than writing a one-off eval script while still being more disciplined than ad hoc prompt tweaking.

## 4. Core concepts

Smarteval should clearly separate four concepts.

### 4.1 Target

The system component being evaluated.

Examples:

- A prompt template
- A Python function that calls an LLM
- A RAG answer-generation function
- An image-generation prompt builder
- A multi-step agent workflow
- A model-based classifier
- A summarization pipeline
- A structured extraction component

The target should be callable by the eval harness.

### 4.2 Levers

The things Smarteval is allowed to modify during experiments.

Examples:

- System prompt
- User prompt template
- Few-shot examples
- Output schema
- Model choice
- Temperature
- Top-p
- Retrieval parameters
- Tool-selection instructions
- Image prompt wording
- Negative prompt
- Post-processing rules

A critical safety rule: Smarteval must not change levers that the engineer has not explicitly allowed.

### 4.3 Scoring vectors

The dimensions used to decide whether a candidate improved the target.

Examples:

- Correctness
- Faithfulness
- Format adherence
- Completeness
- Style match
- Safety
- Latency
- Cost
- Determinism / variance
- Visual quality
- Brand consistency
- Hallucination risk
- User-perceived usefulness

The final score should be decomposable. Smarteval should never hide all decision-making behind a single aggregate number.

### 4.4 Experiment candidates

A candidate is a proposed change to one or more allowed levers.

Each candidate should include:

- Candidate ID
- Name
- Strategy
- Hypothesis
- Exact changes
- Metrics expected to improve
- Potential risks or regressions

Candidates should be auditable and diffable.

## 5. Intended workflow

The Smarteval workflow should follow this loop:

1. Discover the target component.
2. Interview the engineer.
3. Define the objective.
4. Identify allowed levers.
5. Define scoring vectors.
6. Generate the eval plan.
7. Generate or update the eval harness.
8. Run the baseline.
9. Analyze baseline failures.
10. Propose candidate improvements.
11. Run agreed experiments.
12. Compare candidates against baseline and each other.
13. Propose the next iteration.
14. Repeat for the agreed experiment budget.
15. Produce a final report.

The engineer should remain in control of the experiment budget and allowed levers.

---

## 6. Discovery conversation

Smarteval should start with a structured conversation. It should avoid modifying prompts or code before it understands the evaluation goal.

Useful questions include:

- What system behavior do you want to improve?
- Where does this behavior happen in the codebase?
- Is this text generation, image generation, RAG, classification, extraction, or an agent workflow?
- What does a good output look like?
- What are examples of bad outputs?
- Do you already have production traces, test cases, or golden examples?
- What is Smarteval allowed to change?
- What must not change?
- Which matters more: quality, latency, cost, safety, or reliability?
- How many iterations should be run?
- How many candidates should be tested per iteration?
- Should the final result be proposed as a patch, a report, or both?

The output of discovery should be an eval plan.

## 7. Eval plan artifact

The eval plan is the contract between the engineer, Smarteval, and the coding agent.

It should normally live at:

```text
.smarteval/evals/<eval_name>/eval.yaml
```

Example shape:

```yaml
name: support_ticket_summarizer

objective:
  description: Improve factuality, conciseness, and action-item extraction for support ticket summaries.

target:
  type: python_function
  entrypoint: app.services.summarizer:generate_summary

inputs:
  dataset: .smarteval/evals/support_ticket_summarizer/dataset.jsonl

allowed_levers:
  - system_prompt
  - user_prompt_template
  - few_shot_examples
  - temperature

fixed_constraints:
  - output_schema_must_not_change
  - model_must_remain_current_default
  - summary_must_stay_under_150_words

scoring_vectors:
  factual_accuracy:
    type: llm_judge
    weight: 0.35
  action_item_coverage:
    type: llm_judge
    weight: 0.25
  format_adherence:
    type: deterministic
    weight: 0.20
  conciseness:
    type: deterministic
    weight: 0.10
  latency:
    type: runtime
    weight: 0.05
  cost:
    type: runtime
    weight: 0.05

experiment_budget:
  iterations: 3
  candidates_per_iteration: 3
```

This is only an illustrative schema. The implementation may evolve, but the artifact should remain human-readable.

## 8. Repository artifact structure

Smarteval should use explicit repo-local artifacts so both humans and coding agents can inspect the process.

Recommended structure:

```text
.smarteval/
  evals/
    <eval_name>/
      eval.yaml
      dataset.jsonl
      candidates/
        baseline.yaml
        iteration_1_candidate_a.yaml
        iteration_1_candidate_b.yaml
      evaluators/
        factual_accuracy.py
        format_adherence.py
      runs/
        <run_id>/
          results.jsonl
          summary.json
          traces/
      reports/
        final_report.md
```

The goal is to make every step reviewable:

- What was tested
- What changed
- What data was used
- Which metrics were applied
- Which candidate won
- Which regressions appeared

## 9. Evaluator types

Smarteval should support multiple evaluator types.

### 9.1 Deterministic evaluators

Used for hard constraints.

Examples:

- Valid JSON
- Matches schema
- Contains required fields
- Does not exceed word limit
- Does not include forbidden phrases
- Includes required citations
- Uses expected enum values

These should be preferred whenever the rule can be expressed deterministically.

### 9.2 Reference-based evaluators

Used when golden examples or expected outputs exist.

Examples:

- Expected label comparison
- Field-level matching
- Exact match
- Semantic similarity
- Required fact coverage

### 9.3 LLM-as-judge evaluators

Used for subjective or semantic quality.

Examples:

- Helpfulness
- Faithfulness
- Tone
- Completeness
- Brand fit
- Reasoning quality
- Image prompt quality

LLM-as-judge evaluators should use clear rubrics and should ideally produce both a score and a rationale.

### 9.4 Human review evaluators

Used when quality is subjective, high-risk, or business-critical.

Examples:

- Human preference ranking
- Pass/fail annotation
- Pairwise comparison

### 9.5 Runtime evaluators

Used for engineering trade-offs.

Examples:

- Latency
- Token usage
- Cost
- Error rate
- Timeout rate
- Retry count

Runtime metrics should be shown alongside quality metrics to avoid optimizing blindly for quality at unacceptable cost.

## 10. Candidate generation principles

Smarteval should generate candidates based on hypotheses, not random prompt tweaks.

Good candidate strategies include:

- Constraint-first prompt
- Few-shot examples
- Explicit output checklist
- Self-verification step
- Stricter output schema
- Lower-temperature configuration
- Retrieval context narrowing
- Improved system-role framing
- Better negative instructions
- Decomposition into smaller steps

Each candidate should say:

- What changed
- Why it changed
- Which metric should improve
- What could regress

Example candidate:

```yaml
id: iteration_1_candidate_b
name: Few-shot plus output checklist
strategy: examples_and_self_check

hypothesis: >
  Adding two high-quality examples and a final checklist should improve format adherence
  and factual accuracy without materially increasing latency.

changes:
  - Added two few-shot examples to the summarization prompt.
  - Added a pre-output checklist for required fields.
  - Kept model and schema unchanged.

expected_improvement:
  - factual_accuracy
  - format_adherence

risk:
  - higher token usage
  - slightly longer latency
```

## 11. Scoring and comparison

Smarteval should compare candidates at multiple levels:

- Overall weighted score
- Per-metric score
- Regression analysis
- Failure cluster analysis
- Cost and latency impact
- Variance across repeated runs where useful

A report should be able to explain changes like:

```text
Candidate B improved overall score from 72.4 to 81.6.

Main gains:
- factual accuracy: 68 to 84
- format adherence: 80 to 96

Regressions:
- latency: 1.2s to 1.8s
- cost: +18%
```

The winning candidate should not always be the one with the highest aggregate score. The report should mention trade-offs and constraints.

---

## 12. Baseline-first rule

Smarteval should always establish a baseline before proposing or applying improvements.

The baseline answers:

- How good is the current system?
- Which metrics are weak?
- Which examples fail?
- What regressions should future candidates avoid?

A coding agent using Smarteval should not change prompts, model parameters, or retrieval settings before a baseline exists, unless the user explicitly instructs otherwise.

---

## 13. Avoiding eval overfitting

Smarteval should avoid optimizing too tightly against a small eval set.

Recommended dataset split:

- Development set: used for iteration
- Validation set: used to compare candidates
- Holdout set: used only for final confirmation

At minimum, the final report should mention whether a holdout set was used.

Smarteval should flag risks such as:

- Too few examples
- Overuse of LLM judge without deterministic checks
- Candidate optimized only for one metric
- Prompt changes tailored too closely to known examples
- Missing edge cases

## 16. Design constraints

Smarteval should follow these constraints:

- Be project-agnostic.
- Be repo-local by default.
- Keep artifacts human-readable.
- Preserve the baseline.
- Make candidate changes isolated and diffable.
- Never optimize only for the aggregate score.
- Report regressions explicitly.
- Keep the engineer in control of allowed levers and experiment budget.
- Prefer deterministic evaluators when possible.
- Use LLM judges only with explicit rubrics.
- Produce a final report that can be reviewed in a PR.

---

## 17. What a coding agent should do when asked to work on Smarteval

When working on the Smarteval codebase itself, an AI coding agent should:

1. Preserve the separation between planning, execution, analysis, and reporting.
2. Prefer explicit file artifacts over hidden in-memory state.
3. Keep the CLI as the stable primary interface.
4. Keep schemas readable and versionable.
5. Make generated harness code easy for engineers to inspect and edit.
6. Avoid hard-coding assumptions about a specific project structure.
7. Support incremental adoption: one eval, one target, one dataset.
8. Write tests around evaluator behavior and scoring aggregation.
9. Avoid making automatic production changes without an explicit user decision.
10. Keep output reports clear enough for engineering review.

---

## 18. What a coding agent should avoid

An AI coding agent should avoid:

- Treating Smarteval as a generic benchmark runner only.
- Changing prompts before a baseline exists.
- Mixing evaluator changes and target changes in the same candidate without noting it.
- Optimizing only for LLM-judge scores.
- Ignoring latency, cost, safety, or format regressions.
- Creating opaque experiment state that cannot be reviewed.
- Assuming the highest aggregate score is automatically the best choice.
- Generating large, complex frameworks before the MVP loop works.
- Duplicating integration instructions that belong in `smarteval_claude_code_codex_integration_plan.md`.
